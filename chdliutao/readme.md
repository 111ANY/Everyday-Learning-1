* 运动会轻应用总结
```
运动会期间，易班上线了运动会轻应用，主要功能包括：实时查看各个学院的排名和积分，其次可以在易班上对
自己支持的学院点赞打call，也可以查看运动会三天的比赛日程。
参与人员与分工说明：刘涛负责轻应用的后台以及后端逻辑的开发，何卫东负责前端界面的实现，其中点赞的功能是由杨老师开发完成。
技术架构说明：dophinphp的框架。
使用情况说明：运动会轻应用上线于4.17，中途出现了两次bug，第一次为赛程显示时原本显示time的位置显示了name，经过排查为代码字母拼写错误，第二次为周五上午的未开始赛程显示错误，原因是数组下标错误，同时，由于本地的数据库和服务器的数据库中的数据不同，导致本地测试环境与正式环境不同。应用访问量第一天4260，第二天6647，第三天8000+。
本次轻应用所暴露的问题有：
1、没有考虑在移动端显示界面时的宽度问题，导致中途有些名字比较长的赛程无法显示完全。
2、本次轻应用的界面在向手指触碰下滑动时会出现卡顿的情况，猜测是js的问题，这一情况需注意。
3、本次一个很大的问题是成绩需要实时输入，体育办公室贴出来一份成绩，这边得手动输入对应的成绩，起初后台输入是一名一名输入，即一次比赛需要输入8次，4.19日下午对其进行改进，一次输入项目类型id和项目id之后便可输入多个学院的名次以及对应的成绩，但是由于在开发过程中使用的三个表联查的方式，致使输入成绩时必须输入对应的id，增加了出错概率，之后改进时可以考虑直接输入名称不用表之间的关联。
经验启示说明：
1、	对于数据库的设计，宁可增加其长度，不能使之太宽，否则不容易进行后续操作。
2、	在点赞时可以获取对方的IP，从而增加对IP访问的次数限制，这里的数据表设计同样也是一个IP点赞一次一条记录。
3、	在后台开发时，应该以用户操作简单，效率高为主要目的，不可以因为省代码致使输入成绩的效率极低，增加了劳动量。
4、	后台至前台的渲染时，应该考虑更加简洁的方法，而非构造出多维数组这样不利于维护，代码也不整洁的方式。
```
